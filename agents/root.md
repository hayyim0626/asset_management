## 프로젝트 진행 Flow

아래 절차는 새로운 기능 추가, 테스트 코드 작성, 리팩터링 등 프로젝트 진행 시 기본 흐름으로 사용합니다.

### Flow Override

- 사용자가 프롬프트 시작 부분에 `NO_AGENT_FLOW:` 를 명시하면, 아래 기본 진행 Flow는 해당 요청에 한해 적용하지 않습니다.
- 이 경우 브레인스토밍, 문서화, 다중 에이전트 협업, 테스트 반복 절차를 건너뛰고 요청 작업을 직접 수행할 수 있습니다.
- 단, `git-commit-push` 관련 규칙과 사용자 승인 규칙은 그대로 유지합니다.

### 1. 브레인스토밍 먼저 진행

- 코드를 바로 작성하지 않고 먼저 사용자와 충분히 브레인스토밍을 진행합니다.
- 기능 목표, 제약, 화면 흐름, 데이터 흐름, 성공 기준을 먼저 정리합니다.
- 필요하면 관련 plugin, skill, sub-agent를 사용해 구현 사항을 상세하게 논의합니다.
- 브레인스토밍 결과가 충분히 구체화되기 전에는 구현을 시작하지 않습니다.

### 2. 브레인스토밍 결과를 문서로 고정

- 브레인스토밍이 충분히 완료되면 결과를 문서로 정리합니다.
- 새로운 기능 추가인 경우 `docs/feature/{YYYY-MM-DD}{feature-name}/` 디렉토리를 만들고 아래 파일을 생성합니다.
- `prd.md`
- `implements.md`
- 테스트 코드 작성, 리팩터링, 성능 개선, 인프라 정리 등 엔지니어링 성격의 작업은 `docs/engineering/{YYYY-MM-DD}{topic-name}/` 디렉토리를 만들고 아래 파일을 생성합니다.
- `{topic-name}-implements.md`

### 3. 새로운 기능은 5개 에이전트 협업으로 문서를 작성

- 새로운 기능 작업은 총 5개의 에이전트를 사용해 문서를 작성합니다.
- 문서가 확정되기 전에는 코드 구현을 시작하지 않습니다.

#### 3-1. `prd.md` 작성

- `prd.md`는 "무엇을 만들 것인가" 관점으로 작성합니다.
- 20년 경력의 PM 에이전트와 20년 경력의 UI/UX 엔지니어 에이전트가 협업합니다.
- 두 에이전트의 협업 결과로 작성된 `prd.md`는 최종적으로 사용자 컨펌을 받아야 합니다.

#### 3-2. `implements.md` 작성

- `implements.md`는 확정된 `prd.md`를 기반으로 "어떻게 만들 것인가" 관점으로 작성합니다.
- 30년 경력의 독립적인 엔지니어 에이전트 3명이 협업합니다.
- 역할은 백엔드, 프론트엔드, 데브옵스입니다.
- 협업 결과로 작성된 `implements.md`는 최종적으로 사용자 컨펌을 받아야 합니다.

### 4. 문서 컨펌 이후 코드 작성

- `prd.md`와 `implements.md`가 사용자에게 컨펌된 이후에만 실제 코드 작성을 진행합니다.
- 구현 중 문서와 실제 코드가 어긋나면 먼저 문서를 갱신하고 다시 컨펌을 받습니다.

### 5. 코드 작성 이후 테스트 코드 작성

- 코드 구현이 완료되면 테스트 코드를 작성합니다.
- 테스트 코드는 20년 경력의 테스트 전문가(QA) 에이전트가 작성합니다.
- 테스트 범위는 변경된 기능, 회귀 가능성이 있는 경로, 실패 케이스를 포함해야 합니다.

### 6. 테스트 실행은 3개 서브 에이전트 병렬 검증

- 테스트 코드 작성이 끝나면 3개의 서브 에이전트가 병렬로 테스트를 실행합니다.
- 작업 단위를 나누어 실행하지 않고, 각 에이전트가 전체 테스트 코드를 각각 실행합니다.
- 테스트 실행 중 오류가 발견되면 5번으로 돌아가 테스트 코드를 수정하거나 보강합니다.
- 테스트 통과율이 99%에 도달할 때까지 이 과정을 반복합니다.

### 7. 테스트 완료 후 `git-commit-push`

- 테스트 코드 작성과 테스트 실행이 끝난 뒤 `git-commit-push`를 실행합니다.
- 커밋 메시지와 커밋 범위는 사용자 최종 컨펌을 받은 뒤에만 확정합니다.
- 관련 없는 변경은 분리하고, 승인된 범위만 커밋 및 push 합니다.
