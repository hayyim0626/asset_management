# FX Provider Date Fix

## 배경

- 현재 환율 배치는 한국시간 오전 9시 전후에 `/api/cron/market-sync`를 통해 실행된다.
- 환율 공급자는 Frankfurter `latest` 엔드포인트이며, 이 값은 배치 실행 시각의 "당일 실시간 환율"이 아니라 공급자가 제공하는 "최신 영업일 기준 환율"이다.
- 현재 `currency_prices`에는 배치 적재 시각(`created_at`)만 남고, 공급자 기준일은 저장되지 않는다.
- 그 결과 주말 또는 영업일 전환 구간에서 `created_at`만 보면 "오늘 저장됐는데 왜 어제 환율인가"라는 혼동이 발생한다.

## 문제 정의

- 현재 데이터 자체는 대체로 맞지만, 환율의 기준일을 설명할 메타데이터가 없어 운영과 사용자 관점에서 오해가 생긴다.
- 동일한 환율이 여러 날짜에 반복 적재되는 경우, 이것이 공급자 특성 때문인지 배치 이상 때문인지 DB만 보고는 구분하기 어렵다.

## 목표

- `currency_prices`에 환율의 공급자 기준일을 함께 저장한다.
- 배치 응답과 테스트가 공급자 기준일을 명시적으로 다루도록 수정한다.
- 필요한 읽기 계약에 공급자 기준일을 노출할 수 있는 기반을 마련한다.
- 기존 환율/코인 읽기 계약은 깨지지 않도록 유지한다.

## 비목표

- 환율 공급자를 Frankfurter에서 다른 업체로 교체하지 않는다.
- 코인 가격 공급자 구조는 이번 작업에서 변경하지 않는다.
- 실시간 FX 시세 제공으로 요구사항을 바꾸지 않는다.

## 구현 범위

### 1. DB 스키마

- `currency_prices`에 `provider_date date not null` 컬럼을 추가한다.
- 기존 행은 `created_at at time zone 'utc'`의 날짜값으로 백필한다.
- 새 인덱스는 이번 작업에서 필수는 아니며, 조회 요구가 생기면 후속으로 추가한다.

### 2. 동기화 코드

- Frankfurter fetch 결과에서 `date`를 반드시 파싱한다.
- `MarketFxRateRow`에 공급자 기준일 필드를 추가한다.
- `CurrencyPriceInsertRow`에도 같은 필드를 포함해 insert 한다.
- `syncMarketData()` 반환값에 환율 공급자 기준일 요약을 추가한다.

### 3. 읽기 계약

- 현재 UI가 바로 사용하지 않더라도, 환율 읽기 경로가 공급자 기준일을 받을 수 있게 타입 확장 여지를 반영한다.
- 기존 필드 이름과 기존 소비 경로는 유지한다.

### 4. 테스트

- Frankfurter 응답 정규화 테스트에 공급자 기준일 검증을 추가한다.
- 배치 insert 테스트에 `provider_date` 저장 검증을 추가한다.
- 주말 응답이 직전 영업일 날짜를 반환하는 상황을 설명 가능한 형태로 유지한다.

## 성공 기준

- 새로 적재되는 `currency_prices` row는 `exchange_rate`, `created_at`, `provider_date`를 함께 가진다.
- 운영자가 `created_at`과 `provider_date`를 구분해 환율 freshness를 설명할 수 있다.
- 기존 코인/환율 읽기 경로는 타입 오류 없이 유지된다.
- 관련 단위 테스트가 통과한다.

## 리스크 및 대응

- 실제 Supabase RPC가 `currency_prices` 스키마를 고정적으로 참조할 수 있다.
  - 대응: 읽기 계약은 깨지지 않는 additive change만 적용한다.
- 빈 마이그레이션 디렉터리 상태라 스키마 변경 이력이 없을 수 있다.
  - 대응: 이번 변경부터 명시적 migration 파일을 추가한다.
- 프로덕션 데이터는 이미 존재하므로 null 허용 추가 후 백필 없이 바로 not null 제약을 걸면 실패할 수 있다.
  - 대응: migration에서 컬럼 추가, 백필, not null 순서로 처리한다.
