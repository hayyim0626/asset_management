# PRD: US Stocks Portfolio Management Phase 1

## 문제 정의

현재 자산 관리 앱은 `stocks` 자산군을 데이터 모델에 포함하고 있지만, 실제 사용자 경험은 아직 미완성 상태다.

- 사용자는 미국주식/미국 ETF를 검색해 자산에 추가할 수 없다.
- 보유 수량, 매수일, 평단가 또는 총매수금액을 기준으로 포지션을 관리하는 흐름이 없다.
- 최신 시세를 기반으로 한 `USD + KRW` 평가금액, 손익금액, 수익률 표시가 없다.
- 시세 조회 실패 시 마지막 성공값을 유지하면서도 stale 상태를 명확히 알리는 UX가 없다.
- 배당금, 투자예수금, USD 현금 같은 실제 투자 문맥의 자산 흐름이 충분히 반영되지 않는다.

사용자는 현재 나무증권 계좌를 사용하지만, 1차 단계에서 중요한 것은 `증권사 계좌 자동 연동`이 아니라 `계좌 연동 없이도 쓸 수 있는 미국주식 수동 포트폴리오 관리 경험`을 먼저 완성하는 것이다.

## 목표

1. 사용자가 미국주식과 미국 ETF를 검색해 자산에 추가할 수 있어야 한다.
2. 사용자가 `보유 수량 + 평단가` 또는 `총매수금액` 기준 중 하나를 선택해 포지션을 기록할 수 있어야 한다.
3. 시스템은 최신 시세를 바탕으로 현재가, 평가금액, 손익금액, 수익률을 `USD + KRW`로 함께 보여줄 수 있어야 한다.
4. 최신 시세 조회 실패 시 마지막 성공값을 유지하되, 상단 노란 배너로 stale 상태를 명확히 알려야 한다.
5. 사용자는 추가 매수, 일부 매도, 전체 삭제, 평균단가/총매수금액 정정 수정을 할 수 있어야 한다.
6. 사용자는 주식 자산 문맥 안에서 투자예수금과 배당금 기록을 함께 관리할 수 있어야 한다.
7. 사용자는 현금 자산 문맥 안에서 KRW 현금과 USD 현금을 관리할 수 있어야 하며, USD 현금은 평균 환율 정보를 볼 수 있어야 한다.

## 제품 결정

### 1. 계좌 연동과 포트폴리오 관리는 분리한다

- 1차의 본질은 `수동 입력 기반 포트폴리오 관리`다.
- 외부 API는 종목 검색과 시세 조회를 보조하는 역할만 맡는다.
- 실제 보유 수량, 원가, 배당, 매도, 현금, 예수금은 우리 서비스 내부 데이터가 진실 원본이다.

### 2. 증권사 API는 provider로 추상화한다

- 사용자는 나무증권 계좌를 쓰지만, 공개 API 사용 가능성은 구현 전 검증이 필요하다.
- 따라서 PRD 레벨에서는 `증권사 API 기반 종목 검색/시세 조회`를 요구사항으로 두고,
- 구현 단계에서는 `검증된 provider 우선 + 후보 provider 교체 가능` 구조를 허용한다.

### 3. 자산 분류는 사용자 관점을 우선한다

- 큰 자산 축은 `stocks`, `crypto`, `cash`를 유지한다.
- `투자예수금`은 1차에서 `stocks` 하위 카테고리로 취급한다.
- `USD 현금`, `KRW 현금`은 `cash` 자산군에 둔다.
- `배당금`은 현금 자산으로 누적되지만, 어느 종목에서 발생했는지 연결 정보는 반드시 기록한다.

### 4. USD 현금 평균 환율은 직접 취득한 USD 기준으로만 계산한다

- USD 현금 평균 환율은 사용자가 환전하거나 직접 입력한 USD 취득 원가 기준으로 계산한다.
- 배당금으로 들어온 USD는 `USD 잔액`에는 포함되지만 `USD 평균 환율 원가` 계산에는 포함하지 않는다.
- 이유는 배당금 유입은 환전으로 취득한 USD가 아니라 현금 유입 이벤트이기 때문이다.

### 5. 일부 매도는 평단 유지 규칙을 따른다

- 일부 매도 시 평균단가는 유지하고 보유 수량만 감소한다.
- 매도 이벤트는 향후 realised P/L 집계를 위해 `날짜`, `월`, `연도` 기준 라벨링이 가능해야 한다.

### 6. 주식은 범용 자산 RPC에 끼워맞추지 않는다

- `stocks`는 기존 `add_asset`, `remove_asset`, `update_average_price` 같은 범용 자산 mutation에 억지로 편입하지 않는다.
- 미국주식/미국 ETF는 별도 거래 원장과 별도 mutation contract를 갖는다.
- 이유는 `BUY`, `SELL`, `DIVIDEND`, `ADJUST_COST_BASIS` 같은 이벤트를 일반 자산 추가/삭제 API로는 충분히 표현할 수 없기 때문이다.
- 1차 구현에서 프런트 입력값은 UI 편의 계층이고, 포지션/평단/valuation/stale 판단의 진실 원본은 stock domain backend다.

## 사용자 스토리

1. 사용자는 미국주식이나 미국 ETF를 검색해 내 자산에 추가할 수 있어야 한다.
2. 사용자는 매수 단가를 주당 기준 또는 총매수금액 기준 중 편한 방식으로 입력할 수 있어야 한다.
3. 사용자는 현재 보유 자산의 현재가, 평가금액, 손익금액, 수익률을 USD와 KRW로 함께 확인할 수 있어야 한다.
4. 사용자는 시세 갱신이 실패해도 마지막 성공값을 계속 볼 수 있어야 하며, 해당 값이 최신이 아니라는 사실도 명확히 알아야 한다.
5. 사용자는 추가 매수, 일부 매도, 삭제, 정정 수정으로 포지션을 관리할 수 있어야 한다.
6. 사용자는 투자예수금, 배당금, USD/KRW 현금까지 포함해 실제 투자 상태에 가깝게 자산을 관리할 수 있어야 한다.

## 기능 요구사항

### 종목 검색 및 추가

1. 시스템은 미국주식과 미국 ETF를 검색할 수 있어야 한다.
2. 검색 결과는 최소한 `심볼`, `종목명`, `시장 또는 ETF 여부`를 보여줘야 한다.
3. 사용자가 검색 결과를 선택하면 자산 추가 모달로 이어질 수 있어야 한다.
4. 자산 추가 시 사용자는 `보유 수량`, `매수일`, `평단가 또는 총매수금액`을 입력할 수 있어야 한다.
5. 입력 방식은 토글 형태로 제공되어야 하며, 사용자가 한쪽 값을 입력하면 다른 값은 자동 계산되어야 한다.
6. 저장 전에 사용자는 최종 계산된 수량/원가 정보를 확인할 수 있어야 한다.

### 저장 계약 및 원장

- 주식 저장은 범용 자산 추가/삭제 RPC가 아니라 stock 전용 mutation contract를 사용해야 한다.
- 시스템은 최소한 `BUY`, `SELL`, `ADJUST_COST_BASIS` 이벤트를 원장 형태로 기록할 수 있어야 한다.
- 시스템은 거래 원장을 바탕으로 현재 포지션, 평균단가, 총원가를 서버에서 집계할 수 있어야 한다.
- 시스템은 시세 메타데이터와 포트폴리오 원장을 분리해 관리할 수 있어야 한다.
- 시스템은 `/assets` 화면에 필요한 valuation, 손익, stale 상태를 stock 전용 read model 로 조회할 수 있어야 한다.
- 시스템은 주식 기능을 위해 기존 범용 자산 backend 에 임시 adapter 나 hidden field 확장을 추가하지 않아야 한다.

### 자산 편집

7. 사용자는 기존 포지션에 추가 매수를 할 수 있어야 한다.
8. 사용자는 일부 매도를 할 수 있어야 한다.
9. 사용자는 포지션을 완전히 삭제할 수 있어야 한다.
10. 사용자는 평균단가 또는 총매수금액을 정정 수정할 수 있어야 한다.
11. 일부 매도는 `평단 유지 + 보유 수량 감소` 규칙을 따라야 한다.
12. 매도 이벤트는 후속 realised P/L 집계를 위해 날짜 기준 정보를 남길 수 있어야 한다.

### 현금 / 배당 / 예수금

13. 시스템은 KRW 현금과 USD 현금을 관리할 수 있어야 한다.
14. 시스템은 USD 현금에 대해 평균 환율 정보를 표시할 수 있어야 한다.
15. 시스템은 투자예수금을 `stocks` 하위 자산으로 관리할 수 있어야 한다.
16. 시스템은 배당금을 현금 유입으로 기록할 수 있어야 한다.
17. 배당금 기록에는 어떤 종목에서 발생한 배당금인지 연결 정보가 포함되어야 한다.

### valuation 및 표시

18. 시스템은 현재가, 평가금액, 손익금액, 수익률을 계산할 수 있어야 한다.
19. 미국 자산은 `USD 원본값`과 `KRW 환산값`을 함께 보여줘야 한다.
20. 손익 색상은 상승 시 초록, 하락 시 빨강, 중립 시 중립색을 사용해야 한다.
21. 목록 반영은 저장 직후 일관되게 재계산되어야 한다.

### stale 시세 및 fallback

22. 최신 시세 조회 실패 시 시스템은 마지막 성공값을 유지해야 한다.
23. 최신 시세 조회 실패 시 시스템은 상단 노란 경고 배너를 노출해야 한다.
24. 경고 배너는 사용자가 `현재 표시 중인 값이 마지막 성공값`이라는 사실을 이해할 수 있게 해야 한다.
25. stale 상태에서도 사용자는 목록과 손익 값을 계속 볼 수 있어야 한다.

## UX 원칙

- 주식 자산은 더 이상 “준비중” 토스트로 막히지 않아야 한다.
- 계산 로직은 서버에서 단일 기준으로 처리하고, 프런트는 표시와 입력 UX에 집중한다.
- 주식 포트폴리오의 진실 원본은 프런트 hidden field 나 기존 범용 자산 RPC가 아니라 stock domain backend 여야 한다.
- 미국 자산은 항상 `USD + KRW` 관점을 함께 제공해야 한다.
- 입력 토글은 한 번에 하나의 진실 원천만 편집 가능해야 한다.
- 시세 실패는 기능 중단이 아니라 `경고 + 마지막 성공값 유지`로 처리한다.

## 성공 기준

1. 사용자가 미국주식/미국 ETF를 검색해 자산으로 등록할 수 있다.
2. 등록된 자산이 `/assets` 목록에 반영된다.
3. 손익과 평가금액이 `USD + KRW`로 일관되게 계산된다.
4. 시세 실패 시 마지막 성공값 유지 + 상단 경고 배너가 동작한다.
5. 추가 매수, 일부 매도, 정정 수정이 제품 규칙과 충돌 없이 가능하다.
6. 배당금, USD 현금, 투자예수금이 1차 범위 안에서 관리 가능하다.
7. 주식 추가/편집/조회 흐름이 기존 범용 자산 RPC가 아니라 stock 전용 backend contract 위에서 동작한다.

## 오픈 질문

1. 미국 종목 검색/시세에 사용할 실제 provider를 무엇으로 확정할지 구현 단계에서 검증이 필요하다.
2. 나무증권 API가 검색/시세 용도로 실제 사용 가능한지 추가 확인이 필요하다.
3. 1차 손익 표시에 세금/수수료를 완전히 제외할지 여부를 구현 문서에서 명확히 해야 한다.
